GitHub-flow

A simple process:

  1. Branch
  2. Make changes
  3. Pull Request
  4. Discuss
  5. Merge

Branch

First we create a branch for what we are going to change.

git branch <name>
git checkout <name>

A note on branch names

Branches for bugfixes should be named fix/<name> and branches for new features should be names feature/<name>

You should name branches like you name functions. Short sweet and to the point

Make changes

Lots of code!

Don't forget to commit!

It is generally better to commit a lot. It is better if you have a lot of small commits than a few large ones.

This initiates the discussion about the new feature or bugfix.

Pull request

You don't have to wait until you are finished. You can show off your work early, and simply mark it with [WIP] to show it shouldn't be merged yet.

Discussion

When a Pull Request is created, people can view the changes and comment on them. This way, we can easily review each other's code (peer review).

Merge

When a Pull Request is ready, it can be merged into the master branch of the repository.

Note that in general, the person that created (a part of) the pull request shouldn't merge it.

That ensures that we do peer review.

Unfortunately, nobody can really check my code at the moment. But as soon as that is possible, other people should merge my code.

MVC

Don't focus too much on the image. It explains kinda how MVC works, but don't pin anything on that description.

MVC

MVC consists of three parts:

  • Model
  • View
  • Controller
I'll explain all of them.

Model

Models represent the internal data of an application.

For the GEWIS website, think about users, members, activities, etc. etc.

Model - example


<?php
/**
 * A simple user model
 */
class User {
    protected $lidnr;
    protected $email;
    protected $password
    // and getters / setters
}
A simple user model. This class represents one user.

View

The view determines how the given data is shown to the user.

It has some variables (usually models) given by the controller.

View - Example


Your e-mail address is <?= $this->user->getEmail() ?>.
Not really interesting. We don't have to bother much with them anyways, thats what the designers will do.

Controller

Controllers bring everything together. They obtain models and hand them to the view.

Controller - Example


<?php
class UserController {
    public function indexAction()
    {
        $user = new User();
        $user->setEmail('kokx@gewis.nl');
        return array('user' => $user);
    }
}

This simply gives a user with an email address to the view.

You might have noticed already, but I omitted the ending php tag. You can do that, and should do that. It can only give extra (unwanted) whitespace in the output.

Data Mapper

The data mapper forms the bridge between the Models and the database.

Mappers request data from the database, put the data into Models and return the Models.

Still, we are missing something

Database communication is fixed, but there is still some logic to separate.

Service Layer

The service layer is a layer between the controllers and the data, that handles application logic.

Services provide some service to the controllers. This makes the controllers more simple. And separates application logic from the controller logic.

Application logic: Handling data, and what do do with it.
Controller logic: Handling POST data, URL parameters etc.
